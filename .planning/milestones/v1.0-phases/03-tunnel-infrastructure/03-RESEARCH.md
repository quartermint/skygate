# Phase 3: Tunnel Infrastructure - Research

**Researched:** 2026-03-23
**Domain:** WireGuard VPN tunnel + policy-based routing + Docker server deployment
**Confidence:** HIGH

## Summary

Phase 3 connects the Pi to a remote server via an encrypted WireGuard tunnel, extending Phase 1's nftables foundation with a second fwmark (0x2) for non-bypass traffic that routes through wg0 instead of directly to Starlink. The Pi side requires wireguard-tools, a wg0.conf Jinja2 template, extensions to the existing nftables.conf.j2 (tunnel mark, wg0 forwarding, wg0 masquerade), new policy routing rules (table 200 via wg0), CAKE on wg0, and a tunnel health monitor. The server side is a Docker Compose stack running `linuxserver/wireguard` (ARM64-compatible) with forwarding and masquerade for tunneled traffic.

The critical technical challenge is tunnel resilience over Starlink's LEO satellite link -- handoffs every ~15 seconds, CGNAT that drops idle UDP mappings, and variable latency. WireGuard handles this well natively (stateless UDP, auto re-handshake), but routing table management during outages requires a dedicated monitor that falls back to direct routing when the tunnel is unreachable for >2 seconds.

**Primary recommendation:** Build the WireGuard Ansible role following the exact same pattern as the existing `routing/` role (template config, deploy binary, systemd service), extend nftables.conf.j2 with tunnel marks and wg0 rules, add a lightweight Go tunnel-monitor daemon using the bypass-daemon as the code template, and create a server/ Docker Compose stack with linuxserver/wireguard.

**Recommendation on discretion items:**

1. **AllowedIPs:** Use `AllowedIPs = 0.0.0.0/0` with `Table = off` in wg0.conf. This tells WireGuard to accept all traffic destined for the peer but disables wg-quick's automatic routing. 2. **Tunnel monitor:** Standalone Go binary (`cmd/tunnel-monitor/`). Follows the bypass-daemon pattern (build tags, YAML config, systemd service). Separation of concerns -- different failure domains, independent restart.

3. **Health check thresholds:** Handshake timeout of 180 seconds (3 minutes) before fallback. With PersistentKeepalive=25, a healthy tunnel re-handshakes every ~2 minutes. 180s provides margin. Recovery detection: handshake age < 30 seconds restores tunnel routing.

4. **Docker Compose structure:** Single wireguard service for Phase 3. Add a `# Phase 4: proxy service goes here` comment placeholder. This keeps the docker-compose.yml simple now while signaling the expansion point.

5. **ip rule priorities:** Priority 100 for fwmark 0x1 (bypass, table 100) -- already exists. Priority 200 for fwmark 0x2 (tunnel, table 200). Lower number = higher priority, so bypass always wins.

6. **Server-side firewall:** Use linuxserver/wireguard's built-in PostUp/PostDown for iptables NAT masquerade within the container. The Docker host firewall is the operator's responsibility (documented, not automated).

7. **Fallback implementation:** Switch to direct routing by temporarily removing the ip rule for fwmark 0x2 -> table 200. Without the rule, marked traffic falls through to the main routing table and goes out eth0 directly. Restoring the rule re-enables tunnel routing. This is simpler and more predictable than manipulating routes inside table 200.

### Pattern 1: Policy-Based Routing with Dual fwmark

**What:** Extend Phase 1's single fwmark (0x1 bypass) with a second mark (0x2 tunnel) in the same nftables prerouting chain. Traffic matching bypass_v4 set gets 0x1 (table 100, direct via eth0). All other AP-originated traffic gets 0x2 (table 200, via wg0). Connection tracking preserves marks across packet flows.

**When to use:** Any gateway that needs to split traffic between two or more upstream paths based on destination.

**Implementation (nftables.conf.j2 extension):**
```nft
chain prerouting {
    type filter hook prerouting priority mangle;

    # Aviation bypass: direct routing via Starlink (fwmark 0x1)
    ip daddr @bypass_v4 meta mark set 0x1
    ip daddr @bypass_v4 ct mark set meta mark

    # Tunnel: non-bypass AP traffic via WireGuard (fwmark 0x2)
    iifname "{{ ap_interface }}" ip daddr != @bypass_v4 meta mark set 0x2
    iifname "{{ ap_interface }}" ip daddr != @bypass_v4 ct mark set meta mark

    # Restore marks for existing connections
    ct mark != 0x0 meta mark set ct mark
}
```

**Policy routing setup (Ansible task):**
```bash
# Table 100: bypass (existing from Phase 1)
ip rule add fwmark 0x1 table 100 priority 100
ip route add default dev eth0 table 100

# Table 200: tunnel (new in Phase 3)
ip rule add fwmark 0x2 table 200 priority 200
ip route add default dev wg0 table 200
```

### Pattern 2: WireGuard wg0.conf with Table=off for Custom Routing

**What:** Set `Table = off` in the WireGuard client config so wg-quick does not install its own routing rules. All routing is handled by nftables fwmark + ip rule policy routing. This prevents conflicts between wg-quick's automatic routing and our custom split.

**When to use:** Whenever you need custom policy routing with WireGuard instead of the default "route everything through tunnel" behavior.

**Implementation (wg0.conf.j2):**
```ini
[Interface]
PrivateKey = {{ wg_private_key }}
Address = {{ wg_client_address }}/24
MTU = 1420
Table = off

PostUp = ip rule add fwmark 0x2 table 200 priority 200; ip route add default dev wg0 table 200
PostDown = ip rule del fwmark 0x2 table 200 priority 200; ip route del default dev wg0 table 200

[Peer]
PublicKey = {{ wg_server_public_key }}
Endpoint = {{ wg_server_endpoint }}:{{ wg_server_port }}
AllowedIPs = 0.0.0.0/0
PersistentKeepalive = 25
```

**Key insight:** `AllowedIPs = 0.0.0.0/0` tells WireGuard to accept return traffic for any IP (necessary since tunneled traffic goes to arbitrary internet destinations). `Table = off` prevents wg-quick from adding a default route that would capture ALL traffic. The fwmark-based policy routing handles the split.

### Pattern 3: Tunnel Health Monitor with Routing Fallback

**What:** A daemon that checks WireGuard handshake recency via `wg show wg0 latest-handshakes` and manipulates ip rules to fall back to direct routing when the tunnel is unreachable. DNS filtering (Pi-hole) continues regardless.

**State machine:**
```
                      ┌──────────────┐
                      │   HEALTHY    │
                      │ (tunnel up)  │
                      └──────┬───────┘
                             │ handshake age > 180s
                             v
                      ┌──────────────┐
                      │  DEGRADED    │
                      │ (direct rte) │
                      └──────┬───────┘
                             │ handshake age < 30s
                             v
                      ┌──────────────┐
                      │  RECOVERED   │
                      │ (restoring)  │
                      └──────┬───────┘
                             │ route restored
                             v
                      ┌──────────────┐
                      │   HEALTHY    │
                      └──────────────┘
```

**Fallback mechanism:**
```bash
# Enter degraded mode: remove tunnel routing rule
ip rule del fwmark 0x2 table 200 priority 200
# Traffic with fwmark 0x2 falls through to main table -> eth0 -> Starlink direct

# Recover: restore tunnel routing rule
ip rule add fwmark 0x2 table 200 priority 200
# Traffic with fwmark 0x2 now routes to table 200 -> wg0 -> tunnel
```

**Why this approach:** Removing the ip rule is atomic, instant, and reversible. No route manipulation needed inside table 200. The fwmark still gets applied by nftables, but without a matching rule it falls through to the main table. This makes recovery a single command.

### Pattern 4: linuxserver/wireguard Server Configuration

**What:** Docker Compose stack running the linuxserver/wireguard image in server mode with a single peer (the Pi).

**Implementation (docker-compose.yml):**
```yaml
services:
  wireguard:
    image: lscr.io/linuxserver/wireguard:latest
    container_name: skygate-wireguard
    cap_add:
      - NET_ADMIN
      - SYS_MODULE
    environment:
      - PUID=1000
      - PGID=1000
      - TZ=Etc/UTC
      - SERVERURL=${SERVERURL}
      - SERVERPORT=51820
      - PEERS=skygate-pi
      - INTERNAL_SUBNET=10.13.13.0
      - ALLOWEDIPS=0.0.0.0/0
      - PERSISTENTKEEPALIVE_PEERS=all
    volumes:
      - ./wireguard-config:/config
      - /lib/modules:/lib/modules

      - 51820:51820/udp
    sysctls:
      - net.ipv4.conf.all.src_valid_mark=1

```

**Key details:**
- `PEERS=skygate-pi` generates a single peer config with name "skygate-pi"
- Peer config output at `./wireguard-config/peer_skygate-pi/` includes wg0.conf, privatekey, publickey, and QR code PNG
- Server public key at `./wireguard-config/server/publickey`
- `PERSISTENTKEEPALIVE_PEERS=all` sets PersistentKeepalive=25 on all peers
- `ALLOWEDIPS=0.0.0.0/0` allows the Pi to route all tunnel traffic through the server
- `net.ipv4.conf.all.src_valid_mark=1` required for WireGuard routing

### Anti-Patterns to Avoid

- **Using wg-quick's default routing (Table = auto):** With `AllowedIPs = 0.0.0.0/0`, wg-quick creates a routing table that captures ALL traffic. This breaks the split-tunnel architecture. Always use `Table = off` with custom policy routing.

- **Running WireGuard tunnel before nftables is ready:** If wg0 comes up before nftables rules are loaded, traffic can leak unfiltered. systemd ordering: `After=nftables.service`.

- **Monitoring tunnel by pinging through wg0:** Active pinging wastes satellite bandwidth and adds latency. Passive monitoring via `wg show wg0 latest-handshakes` is free -- WireGuard already does handshakes every ~2 minutes with PersistentKeepalive.

- **Manipulating routes inside table 200 for fallback:** Adding/removing routes inside the table is more complex and error-prone than adding/removing the ip rule itself. Rule manipulation is atomic and has no edge cases.

## Common Pitfalls

### Pitfall 1: WireGuard wg-quick Routing Conflict
**What goes wrong:** wg-quick with `AllowedIPs = 0.0.0.0/0` and default `Table = auto` creates its own routing table (51820) and fwmark, conflicting with SkyGate's custom policy routing.
**Why it happens:** wg-quick's default behavior assumes it controls all routing. With split tunneling, routing is managed externally.
**How to avoid:** Always set `Table = off` in the [Interface] section of wg0.conf. Manage all routing via nftables fwmark + ip rule.
**Warning signs:** All traffic going through the tunnel (bypass traffic stops working), duplicate routing table entries, `ip rule show` showing unexpected fwmark rules.

### Pitfall 2: Connection Tracking Mark Restoration Order
**What goes wrong:** New connections get the correct fwmark, but subsequent packets in the same flow don't -- they lose their mark and route incorrectly.
**Why it happens:** nftables connection tracking marks (`ct mark`) must be saved on new connections AND restored on subsequent packets. The restore rule must come AFTER the set rules, and it must handle the case where ct mark is already set.
**How to avoid:** Use the pattern: (1) mark new connections with meta mark + ct mark set, (2) restore ct mark to meta mark for existing connections. The restore rule uses `ct mark != 0x0` to avoid overwriting active marks.
**Warning signs:** First request to a domain works (gets tunnel mark), subsequent requests to the same domain route direct (lost mark).

### Pitfall 3: Starlink CGNAT Drops Idle Tunnel
**What goes wrong:** WireGuard tunnel silently dies after ~2 minutes of inactivity. No error message, no reconnection attempt. Traffic blackholes.
**Why it happens:** Starlink's CGNAT aggressively drops idle UDP mappings. Without keepalive, the NAT mapping expires and inbound packets from the server can't reach the Pi.
**How to avoid:** `PersistentKeepalive = 25` in the [Peer] section. This sends a keepalive packet every 25 seconds, keeping the NAT mapping alive. Already locked in D-03.
**Warning signs:** Tunnel works initially but fails after a few minutes of no traffic. `wg show wg0` shows `latest handshake` growing past 3+ minutes.

### Pitfall 4: MTU Misconfiguration Causing Silent Packet Loss
**What goes wrong:** Large packets (close to 1500 bytes) are silently dropped because they exceed the effective MTU after WireGuard encapsulation. Small requests (DNS, API calls) work fine; large downloads and page loads fail or stall.
**Why it happens:** WireGuard adds 60-80 bytes of overhead. If the outer link MTU is 1500 (Starlink) and the inner MTU is also 1500, packets that are nearly max-size get encapsulated to >1500 bytes and are fragmented or dropped.
**How to avoid:** Set `MTU = 1420` in wg0.conf (locked in D-02). This is Starlink 1500 minus WireGuard 80-byte overhead. Consider MSS clamping in nftables as defense-in-depth: `tcp flags syn tcp option maxseg size set 1380`.
**Warning signs:** Small requests work, large file downloads stall or are extremely slow. `tcpdump` shows ICMP "fragmentation needed" messages.

### Pitfall 5: PostUp/PostDown Not Executing on systemd-managed wg-quick
**What goes wrong:** `PostUp` and `PostDown` directives in wg0.conf don't execute, leaving policy routing rules missing.
**Why it happens:** When WireGuard is managed via `systemctl start wg-quick@wg0`, the shell environment may differ from manual `wg-quick up wg0`. Path issues or permission problems can silently skip PostUp.
**How to avoid:** Use absolute paths in PostUp/PostDown (`/usr/sbin/ip rule add ...`). Alternatively, manage policy routing in a separate Ansible task with idempotent shell commands (the Phase 1 pattern), not in PostUp/PostDown. This is more robust and easier to debug.
**Warning signs:** `ip rule show` missing the fwmark 0x2 rule after boot despite wg0 being up.

### Pitfall 6: Fallback Monitor Race Condition
**What goes wrong:** Tunnel monitor removes the ip rule for fallback, then WireGuard re-handshakes in the same second, and the monitor restores the rule -- but during the brief window, in-flight TCP connections got RST'd because their route changed mid-stream.
**Why it happens:** The handshake timeout check and rule manipulation are not atomic. Tunnel state can change between check and action.
**How to avoid:** Use hysteresis: require 3 consecutive failed checks before entering degraded mode, and 3 consecutive successful checks before recovering. This prevents flapping on transient handoffs. With 5-second check interval, this means ~15 seconds before state change.
**Warning signs:** Dashboard shows tunnel status toggling rapidly, users experience intermittent 1-2 second freezes.

## Code Examples

### nftables.conf.j2 -- Extended for Phase 3

```nft
#!/usr/sbin/nft -f
# Managed by Ansible - SkyGate Phase 3 nftables
# Do NOT edit manually on the Pi

flush ruleset

table inet skygate {
    # Aviation bypass IP set (populated by Go bypass daemon)
    set bypass_v4 {
        type ipv4_addr
        flags timeout
        timeout 1h
    }

    # Captive portal allowed MACs (populated by dashboard daemon)
    set allowed_macs {
        type ether_addr
        flags timeout
        timeout 24h
    }

    chain input {
        type filter hook input priority filter; policy drop;

        ct state established,related accept
        iif "lo" accept

        # Allow DHCP on AP interface
        iifname "{{ ap_interface }}" udp dport 67 accept

        # Allow DNS on AP interface (to Pi-hole)
        iifname "{{ ap_interface }}" udp dport 53 accept
        iifname "{{ ap_interface }}" tcp dport 53 accept

        # Allow HTTP/HTTPS on AP interface (dashboard + captive portal)
        iifname "{{ ap_interface }}" tcp dport { 80, 443 } accept

        # Allow WireGuard on uplink (return traffic from server)
        iifname "{{ uplink_interface }}" udp dport {{ wg_listen_port | default(51820) }} accept

        # Allow SSH (development only)
        tcp dport 22 accept

        # Allow ICMP
        ip protocol icmp accept
    }

    chain forward {
        type filter hook forward priority filter; policy drop;

        # Allow forwarding from AP to Starlink (bypass traffic)
        iifname "{{ ap_interface }}" oifname "{{ uplink_interface }}" accept

        # Allow forwarding from AP to WireGuard tunnel (tunnel traffic)
        iifname "{{ ap_interface }}" oifname "wg0" accept

        # Allow established/related return traffic
        ct state established,related accept

        # Per-interface byte counters (for Phase 2 monitoring)
        iifname "{{ ap_interface }}" counter
        oifname "{{ ap_interface }}" counter
    }

    chain prerouting {
        type filter hook prerouting priority mangle;

        # Aviation bypass: direct routing via Starlink (fwmark 0x1)
        ip daddr @bypass_v4 meta mark set 0x1
        ip daddr @bypass_v4 ct mark set meta mark

{% if wg_enabled | default(false) %}
        # Tunnel: non-bypass AP traffic via WireGuard (fwmark 0x2)
        iifname "{{ ap_interface }}" ip daddr != @bypass_v4 meta mark set 0x2
        iifname "{{ ap_interface }}" ip daddr != @bypass_v4 ct mark set meta mark
{% endif %}

        # Restore marks for existing connections
        ct mark != 0x0 meta mark set ct mark
    }

    chain postrouting {
        type nat hook postrouting priority srcnat;

        # NAT from AP to Starlink
        oifname "{{ uplink_interface }}" masquerade

{% if wg_enabled | default(false) %}
        # MSS clamping for tunnel traffic (prevent MTU issues)
        oifname "wg0" tcp flags syn tcp option maxseg size set 1380
{% endif %}
    }
}
```

### Tunnel Monitor -- Health Check Logic (Go)

```go
// Source: WireGuard docs + wg-healthcheck pattern
// health.go

package main

import (
    "fmt"
    "strconv"
    "strings"
    "time"
)

// TunnelState represents the current tunnel health state
type TunnelState int

const (
    StateHealthy  TunnelState = iota
    StateDegraded
)

// CheckHandshake parses `wg show wg0 latest-handshakes` output and returns
// whether the tunnel is healthy based on handshake recency.
func CheckHandshake(output string, maxAge time.Duration) (healthy bool, age time.Duration, err error) {
    // Output format: "<public_key>\t<unix_timestamp>\n"
    lines := strings.TrimSpace(output)
    if lines == "" {
        return false, 0, fmt.Errorf("no handshake data (tunnel may not be configured)")
    }

    parts := strings.Fields(lines)
    if len(parts) < 2 {
        return false, 0, fmt.Errorf("unexpected wg show output: %q", lines)
    }

    ts, err := strconv.ParseInt(parts[1], 10, 64)
    if err != nil {
        return false, 0, fmt.Errorf("parsing timestamp: %w", err)
    }

    if ts == 0 {
        // Timestamp 0 means no handshake has ever occurred
        return false, 0, nil
    }

    age = time.Since(time.Unix(ts, 0))
    return age < maxAge, age, nil
}
```

## Open Questions

1. **CAKE bandwidth ceiling for wg0**
   - What we know: eth0 CAKE uses 5-100 Mbps range. WireGuard adds overhead (~5-10%).
   - What's unclear: Optimal wg0 CAKE bandwidth relative to eth0 CAKE. Setting wg0 CAKE higher than eth0 CAKE is wasteful; setting it too low artificially limits tunnel throughput.
   - Recommendation: Set wg0 CAKE to ~75-80% of eth0 CAKE ceiling. The tunnel is bandwidth-limited by the outer link anyway. Start with 15 Mbps base / 80 Mbps max and tune empirically.

2. **Autorate dual-interface coordination**
   - What we know: Existing autorate.sh monitors one interface. D-20 says extend to monitor wg0 alongside eth0.
   - What's unclear: Should there be two independent autorate loops or one coordinated script? Two interfaces measuring latency against the same reflector may conflict.
   - Recommendation: Run a single autorate instance for eth0 (the physical link). Let CAKE on wg0 use a static bandwidth ceiling derived from eth0's current rate. Simpler and avoids measurement interference. The wg0 CAKE prevents bufferbloat within the tunnel; eth0 CAKE prevents bufferbloat on the satellite link.

3. **wg_enabled flag vs always-on tunnel config**
   - What we know: The Pi needs to work with or without WireGuard configured (degraded mode is a feature).
   - What's unclear: Should nftables rules for tunnel marking always be present (with traffic falling through to direct when wg0 is down) or conditionally included via `wg_enabled` Ansible var?
   - Recommendation: Use `wg_enabled` Ansible variable to conditionally include tunnel nftables rules. When false, the Pi operates in Phase 1 mode (all traffic direct). When true, tunnel rules are active. This is cleaner than runtime detection and matches the manual key exchange workflow (D-10).

# Phase 3: Tunnel Infrastructure - Context

**Gathered:** 2026-03-23
**Status:** Ready for planning

<domain>
## Phase Boundary

WireGuard tunnel from Pi to a remote server with policy-based routing -- non-aviation traffic flows through the encrypted tunnel while aviation apps (ForeFlight, Garmin Pilot, weather APIs) continue routing directly to Starlink. Includes remote server WireGuard endpoint setup via Docker Compose. No content proxy (Phase 4), no certificate management (Phase 5), no dashboard changes (Phase 2) -- just the tunnel plumbing and routing rules.

</domain>

<decisions>
## Implementation Decisions

### WireGuard Configuration
- **D-01:** WireGuard kernel module on Pi (built into kernel 6.12, zero userspace overhead). Install `wireguard-tools` for `wg` and `wg-quick` userspace utilities via Ansible.
- **D-02:** MTU set to 1420 for IPv4 (Starlink 1500 - WireGuard 80 byte overhead). Empirical testing recommended during verification but 1420 is the conservative safe default.
- **D-03:** `PersistentKeepalive = 25` is mandatory -- Starlink CGNAT drops idle UDP mappings aggressively. This keeps the tunnel alive through satellite handoffs.
- **D-04:** WireGuard config file templated via Ansible Jinja2 at `/etc/wireguard/wg0.conf` on Pi side. Server-side config generated by Docker Compose setup script.

### Remote Server Deployment
- **D-05:** Docker Compose one-command deployment for the remote server -- `docker compose up -d` stands up WireGuard server endpoint. This phase deploys WireGuard only; the content proxy container is added in Phase 4.
- **D-06:** Server target is any VPS with 1GB+ RAM (Hetzner CX22 recommended: 2 vCPU, 4GB, ~$4.50/mo). Server OS is standard Linux with Docker.
- **D-07:** Use `linuxserver/wireguard` Docker image for the server-side WireGuard endpoint. Mature, well-documented, handles key generation and peer management.
- **D-08:** Single peer configuration (one Pi, one server) for v1. Multi-peer/multi-tenant is deferred to hosted service buildout (out of scope per PROJECT.md).

### Key Management
- **D-09:** Key pair generation happens during first-boot or initial Ansible provisioning. Private keys never leave the device they're generated on. Server public key is configured in Pi's wg0.conf; Pi public key is configured as a peer on the server.
- **D-10:** Key exchange workflow: (1) server generates keys during `docker compose up`, outputs server public key and endpoint; (2) operator pastes server public key + endpoint into Ansible vars; (3) Ansible provisions Pi with wg0.conf containing server info; (4) Pi public key is added to server peer config. Manual for v1 -- automated key exchange deferred.

### Policy-Based Routing (extending Phase 1 nftables)
- **D-11:** Extend existing nftables prerouting chain to add tunnel mark (fwmark 0x2) for non-bypass traffic. Phase 1 already marks bypass traffic with 0x1 -- Phase 3 adds the complementary "everything else goes through tunnel" mark.
- **D-12:** Two ip rule policy routes: `fwmark 0x1 -> table 100` (direct via eth0/Starlink) and `fwmark 0x2 -> table 200` (via wg0 tunnel). Table 100 already exists from Phase 1 routing; table 200 is new.
- **D-13:** Forward chain updated to allow forwarding from AP interface to wg0 (in addition to existing AP-to-eth0 rule). Return traffic via `ct state established,related` already covers the reverse path.
- **D-14:** NAT masquerade added for wg0 outbound in addition to existing eth0 masquerade -- traffic exiting the tunnel to the internet needs source NAT on the server side.

### Tunnel Resilience & Fallback
- **D-15:** Fallback to direct routing if tunnel is unreachable for >2 seconds. DNS filtering (Pi-hole) continues regardless of tunnel state. This is the "degraded mode" from the architecture doc -- seamless, not error-prone.
- **D-16:** Tunnel health monitoring via a lightweight Go daemon or bash script that checks WireGuard handshake recency (`wg show wg0 latest-handshakes`). If last handshake > N seconds, trigger fallback routing.
- **D-17:** Auto-reconnect is built into WireGuard's design (stateless UDP, automatic handshake re-initiation). The fallback monitor's job is routing table management during outages, not tunnel reconnection.
- **D-18:** Dashboard integration deferred -- Phase 3 does not modify the dashboard. Tunnel status can be exposed via a status file or API endpoint for Phase 2 dashboard to consume later.

### CAKE QoS on WireGuard Interface
- **D-19:** CAKE qdisc applied to wg0 interface in addition to the existing eth0 CAKE from Phase 1. Tunnel traffic needs its own shaping to prevent bufferbloat within the encrypted tunnel.
- **D-20:** Autorate script extended to monitor and shape wg0 alongside eth0. Same fping-based latency measurement approach.

### Claude's Discretion
- Exact WireGuard AllowedIPs configuration for split tunneling (likely `0.0.0.0/0` with policy routing handling the split, or explicit subnet exclusions)
- Tunnel health check thresholds (handshake timeout before fallback, recovery detection)
- Whether tunnel monitor is a standalone Go binary, part of the bypass daemon, or a bash script
- Docker Compose structure for server (single container vs multi-service with proxy placeholder)
- ip rule priority values for policy routing tables
- Server-side firewall rules (nftables/iptables on the Docker host)
- Exact fallback routing implementation (remove wg0 default route vs change fwmark rules)

</decisions>

## Existing Code Insights

### Reusable Assets
- **nftables.conf.j2**: Already has `bypass_v4` set, prerouting chain with fwmark 0x1, postrouting NAT. Extend with fwmark 0x2 and wg0 forwarding rules.
- **bypass-daemon Go pattern**: YAML config, platform-specific build tags (`nftset_linux.go` / `nftset_stub.go`), systemd service template. Tunnel monitor can follow same pattern.
- **Ansible role structure**: `routing/`, `networking/`, `qos/` roles exist. New `wireguard` role follows identical structure (tasks/templates/handlers).
- **Makefile**: Cross-compile target (`GOOS=linux GOARCH=arm64 CGO_ENABLED=0`) and Ansible deploy target already established.
- **group_vars/all.yml**: Centralized config for interface names, paths -- extend with WireGuard-specific vars.
- **systemd service template** (`skygate-bypass.service.j2`): Pattern for `After=`, `Requires=`, security hardening, restart behavior.

### Established Patterns
- Go daemons with platform-specific build tags (linux vs stub for macOS dev)
- YAML for config files, Jinja2 for Ansible templates
- Cross-compile: `GOOS=linux GOARCH=arm64 CGO_ENABLED=0`
- nftables for all firewall/routing rules (no iptables)
- systemd services with `After=nftables.service`, `Restart=always`, `ProtectSystem=strict`
- Data persistence in `/data/skygate`, binaries in `/opt/skygate`

### Integration Points
- nftables prerouting chain: add fwmark 0x2 for tunnel-bound traffic (complement to existing 0x1 bypass)
- nftables forward chain: add wg0 as allowed output interface alongside eth0
- nftables postrouting: add masquerade for wg0 (server-side) or confirm Pi-side NAT
- Policy routing: add ip rule for fwmark 0x2 -> table 200 -> default via wg0
- Phase 1 QoS (autorate): extend CAKE to wg0 interface
- Phase 2 dashboard (future): tunnel status endpoint for degraded mode display

<specifics>
## Specific Ideas

- WireGuard is kernel-mode on Pi 5 (kernel 6.12) -- zero userspace overhead, just needs wireguard-tools package
- Starlink CGNAT requires PersistentKeepalive = 25 -- without this, tunnel drops after ~2 minutes of inactivity
- MTU 1420 is the conservative safe value for IPv4 over WireGuard over Starlink -- test empirically but start here
- Fallback routing must be seamless -- passengers should never see a loading spinner because the tunnel dropped during a satellite handoff
- Server-side Docker Compose is Phase 3's only server-side deliverable -- it stands up WireGuard only, no proxy yet
- The degradation chain from ARCHITECTURE.md applies: full system -> degraded (DNS only, direct routing) -> minimal (WiFi only)
- Tunnel monitor should be lightweight -- checking `wg show wg0 latest-handshakes` is cheaper than active probing

</specifics>
